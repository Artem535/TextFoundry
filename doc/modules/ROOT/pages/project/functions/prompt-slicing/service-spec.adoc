= Prompt Slicing Service Spec

== 1. Purpose and Service Boundary

* Service responsibility: разложить source prompt на набор block suggestions
* Consumers: GUI slicing workflow
* Non-goals: direct persistence, deterministic parser for every source format

== 2. Consumers and Dependencies

* Consumers: Block Slice Dialog, composition update workflow
* Dependencies: `IBlockGenerator`, provider API, client-side validation

== 3. API / Tool Methods

* `Engine.GenerateBlockBatchData(...)`
* `Engine.GenerateBlockDrafts(...)`

== 4. Input Schema

* `PromptSlicingRequest.source_text`
* mode: generate or update
* preservation-oriented parameters

== 5. Output Schema

* `GeneratedBlockBatch`
* `std::vector<BlockDraft>`

== 6. Data Contract Rules

* generated ids должны быть уникальны
* output должен быть пригоден к review before apply
* collisions with existing ids must be surfaced

== 7. Prompting and Source Hierarchy

* source prompt is the primary input
* update mode may include context from existing composition
* provider-specific prompt policy lives in generator implementation

== 8. Error Handling and Retries

* provider/transport errors surface to caller
* validation errors stop apply flow
* retries are transport/provider-level, not persistence-level

== 9. Confidence / Routing Rules

* current implementation uses one configured generator path
* preservation heuristics act as a practical quality gate

== 10. Edge Cases and Fallback Behavior

* empty source prompt -> validation failure
* duplicate ids -> reject generated batch
* too-aggressive update -> user reviews and falls back to manual editing
