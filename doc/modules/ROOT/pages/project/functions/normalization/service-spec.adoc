= Normalization Service Spec

== 1. Purpose and Service Boundary

* Service responsibility: apply semantic rewrite to text or composition
* Consumers: render UI, composition UI, future authoring clients
* Non-goals: implicit render-time rewrite, direct mutation of published state

== 2. Consumers and Dependencies

* Consumers: render workflow, composition workflow
* Dependencies: `INormalizer`, `IBlockNormalizer`, provider API

== 3. API / Tool Methods

* `Engine.Normalize(...)`
* `Engine.NormalizeComposition(...)`

== 4. Input Schema

Text normalization input:

* source text
* `SemanticStyle`

Composition normalization input:

* `source_composition_id`
* optional `source_version`
* `SemanticStyle`
* optional `target_composition_id`
* `normalize_static_text`
* `reuse_cached_blocks`

== 5. Output Schema

* normalized string
* `NormalizedCompositionResult`

== 6. Data Contract Rules

* caller must provide explicit style request
* absence of configured normalizer is an error
* derived result is review output, not implicit persistence
* composition normalization preserves composition structure and rewrites block refs

== 7. Prompting and Source Hierarchy

* style policy derives from `SemanticStyle`
* prompt assembly is provider-side implementation detail in `textfoundry_ai`

== 8. Error Handling and Retries

* transport and parsing failures are surfaced
* no hidden retry on persistence because persistence is outside the service boundary

== 9. Confidence / Routing Rules

* current implementation uses configured normalizer path only
* no separate confidence score is exposed to caller

== 10. Edge Cases and Fallback Behavior

* no normalizer configured -> explicit error
* unclear style request -> provider result still requires review
* provider unavailable -> fallback to raw deterministic output
