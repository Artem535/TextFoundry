= Block Generation System Design

== 1. Executive Summary

Функция строит AI-assisted draft generation для одного блока и сохраняет
границу между suggestion и mutation. Генерация идёт через adapter layer,
а публикация и versioning остаются в core authoring workflow.

== 2. Context and Scope

* In scope: single-block generation, provider call, review in GUI block editor
* Out of scope: auto-publish, hidden generation in render path, batch slicing
* Assumption: provider возвращает структуру, пригодную для `GeneratedBlockData`

== 3. Requirements Summary

* Functional: generate block data, convert to draft, show result in editor tab
* NFR: понятные ошибки, opt-in execution, отсутствие влияния на deterministic mode
* UX: generated result must remain editable before save

== 4. Components and Integrations

Компоненты:

* `BlockEditorViewModel`
* `tf::Engine`
* `IBlockGenerator`
* `OpenAiCompatibleBlockGenerator`
* `QtHttpTransport`
* session-level AI configuration in `SessionViewModel`

== 5. Key System Scenario

Поток:

[mermaid]
----
flowchart LR
  User["User Instruction"] --> GUI["GUI Block Editor"]
  GUI --> Engine["Engine.GenerateBlockData / GenerateBlockDraft"]
  Engine --> Generator["IBlockGenerator"]
  Generator --> Provider["OpenAI-compatible API"]
  Provider --> Generator
  Generator --> Engine
  Engine --> GUI["Populate editor fields"]
  GUI --> Save["Manual Save / PublishBlock or UpdateBlock"]
----

== 6. Reliability and Degradation

* при отсутствии generator engine возвращает explicit error
* при ошибке provider пользователь остаётся в ручном authoring workflow
* generated output не должен автоматически записываться в storage
* смена `base_url`, `model` или `api_key` пересобирает engine через session layer

== 7. Operations Summary

* функция explicit и opt-in
* результат сначала попадает в editor workspace
* сохранение новой версии остаётся отдельным действием пользователя
