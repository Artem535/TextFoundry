== Цель документа

Этот документ описывает ограничения, архитектурные принципы, жизненные циклы сущностей и roadmap *Text Engine* — движка для конструирования текстов и промтов из переиспользуемых смысловых блоков.

Документ служит:

* фиксацией инженерных решений
* защитой от расползания требований
* контрактом между Core, Storage, Server и UI
* ориентиром для будущей реализации (CLI / GUI / интеграции)

== Область применения

Text Engine предназначен для:

* сборки текстов из логических блоков
* переиспользования фрагментов между разными текстами
* параметризации и вариативности формулировок
* контролируемой нормализации стиля

Text Engine *не является*:

* LLM-фреймворком
* системой автоматического перевода
* WYSIWYG-редактором

== Ключевые ограничения (Design Constraints)

=== Принцип engine-first

Движок:

* разрабатывается как библиотека
* не зависит от UI, CLI или сетевого протокола
* имеет единый публичный API

Любые интерфейсы считаются *клиентами* движка.

=== Отсутствие "магии"

На ранних этапах движок:

* не выполняет семантическое переписывание текста
* не меняет язык или время автоматически
* не пытается угадывать намерения пользователя

Все преобразования должны быть:

* детерминированными
* воспроизводимыми
* тестируемыми

=== Контроль над текстом важнее удобства

Если автоматизация приводит к:

* потере смысла
* размытию формулировок
* скрытым изменениям текста

— такая автоматизация запрещена.

== Основные сущности

=== Block

Логический переиспользуемый фрагмент текста.

Характеристики:

* уникальный идентификатор
* тип (role, constraint, style, domain, meta)
* шаблон содержания
* параметры
* теги
* язык
* версия
* состояние жизненного цикла

Block не знает:

* где он используется
* в каком порядке
* в каком финальном тексте

=== Composition

Композиция блоков, образующая конкретный текст.

Свойства:

* упорядоченный список Block (с указанием версий)
* локальные параметры
* целевой язык
* целевой стиль
* состояние жизненного цикла

Composition не содержит текст напрямую — только структуру.

=== StyleProfile

Описание требований к финальному тексту.

Может включать:

* язык
* тон
* время (tense)
* форматирование

StyleProfile задаёт *намерение*, но не гарантирует автоматическое приведение текста.

=== Renderer

Компонент, выполняющий:

* подстановку параметров
* разворачивание шаблонов
* сборку итогового текста

Renderer не выполняет:

* перевод
* переформулирование
* смысловую коррекцию

== Жизненный цикл сущностей

Все основные сущности (Block, Composition) имеют явный жизненный цикл.

=== Состояния

* *Draft* — редактируемая версия
* *Published* — зафиксированная неизменяемая версия
* *Deprecated* — устаревшая, но доступная для чтения

=== Правила

* Draft можно изменять без ограничений
* Published версии:
** неизменяемы
** адресуются по версии
** могут использоваться Prompt Server
* Любое изменение Published сущности создаёт новую версию
* Deprecated версии:
** не удаляются
** не используются по умолчанию

== Этапы обработки текста

[source]
----
Blocks
↓
Template Expansion
↓
Raw Text
↓
(optional) Normalization
↓
Final Text
----

=== Template Expansion

Детерминированная стадия:

* подстановка параметров
* разворачивание словарей
* выбор вариантов

=== Normalization (опционально)

Отдельный этап, который:

* может отсутствовать
* может быть реализован через LLM
* может быть отключён полностью

Нормализация никогда не является обязательной.

== Поддержка языков и времени

=== Базовая стратегия

На уровне движка поддерживаются только:

* словари терминов
* шаблонные конструкции
* нейтральные формулировки

=== Что НЕ поддерживается без LLM

* полноценный перевод текста
* автоматическое изменение времени
* сохранение стиля при переписывании

== Storage Model

Storage является инфраструктурным слоем и полностью отделён от бизнес-логики движка.

=== Принципы хранения

* движок не зависит от конкретной БД
* работа идёт только через репозитории
* формат хранения не является частью публичного API

=== Проекты и пространства имён

Минимальная адресация:

* project_key
* entity_type
* entity_id
* version

Разделение по проектам является обязательным требованием для Prompt Server.

=== Read / Write модели

Допускается разделение режимов:

* read-write — UI, редакторы
* read-only — Prompt Server

Prompt Server никогда не работает с Draft-данными.

=== Версионирование

Storage обязан поддерживать:

* хранение версий
* доступ к конкретной версии
* неизменяемость опубликованных сущностей

== Publishing и синхронизация

=== Publishing

Publishing — явная операция, которая:

* фиксирует текущую Draft-версию
* создаёт Published-версию
* делает её доступной для Prompt Server

Publishing является необратимым действием.

=== Sync модель

* UI работает с Workspace (Draft)
* Prompt Server читает только Published Storage
* Обновление промта = публикация новой версии
* Старые версии продолжают обслуживаться

'''

== 🔧 CLI и Terminal UI (обязательный интерфейс)

=== CLI (обязательный компонент)

CLI является **обязательным интерфейсом** системы и рассматривается как *reference client* для Text Engine.

CLI предназначен для:

* проверки полноты и удобства Core API
* инженерного использования без GUI
* автоматизации (скрипты, CI, dev workflows)
* публикации и инспекции промтов

CLI — это **не временный инструмент**, а архитектурный якорь проекта.

==== Роль CLI в архитектуре

CLI:

* является каноническим способом взаимодействия с Core API
* использует те же абстракции, что и Prompt Server / UI
* не содержит бизнес-логики
* не обходит ограничения движка

Любая функциональность, добавляемая в Core, **должна быть достижима через CLI**.

Если операция невозможна через CLI — это архитектурная проблема.

==== Назначение CLI

CLI используется для:

* управления блоками
* управления композициями
* рендеринга текста
* работы с версиями
* публикации промтов
* диагностики и инспекции структуры

CLI **не предназначен** для:

* визуального редактирования сложных структур
* WYSIWYG-работы
* замены GUI

==== Примерные сценарии использования

[source]
----
tf block list
tf block show role.system
tf block render role.system --lang=en

tf composition inspect finance_advisor
tf composition render finance_advisor --version=1.2

tf prompt publish finance_advisor
tf prompt diff finance_advisor --from=1.1 --to=1.2
----

Командный интерфейс считается стабильным API для инженеров.

'''

=== Terminal UI (TUI) — допустимое расширение

Terminal UI (TUI) допускается как **надстройка над CLI**, но не как отдельный интерфейс с собственной логикой.

TUI может быть реализован с использованием:

* ftxui
* termox2

Выбор конкретной библиотеки не фиксируется в PRD.

==== Назначение TUI

TUI предназначен для:

* удобной навигации по структуре блоков
* просмотра композиций
* инспекции зависимостей
* быстрого предпросмотра рендеринга

TUI **не заменяет**:

* Qt/QML UI
* Core API
* CLI

==== Архитектурные ограничения TUI

TUI:

* использует CLI или Core API напрямую
* не содержит логики рендеринга
* не управляет версиями напрямую
* не имеет собственного слоя хранения

TUI рассматривается как *UX-обёртка* над CLI, а не как самостоятельный клиент.

==== Статус реализации

TUI:

* не является обязательным для MVP
* может быть реализован после стабилизации CLI
* не влияет на критерии успешности проекта

Отсутствие TUI **не блокирует** развитие Prompt Server или UI.

== Prompt Server

=== Назначение

Prompt Server — сервис дистрибуции и разрешения промтов.

Он:

* не редактирует данные
* не генерирует новые версии
* не содержит логики рендеринга

=== Запрос клиента

Клиент передаёт:

* project_key
* prompt_name (или путь)
* параметры (опционально)

=== Ответ сервера

Сервер возвращает:

* финальный текст
* метаданные:
** версия
** язык
** стиль

=== Внутренняя логика

Prompt Server:

* разрешает Composition по имени
* загружает зафиксированные версии Block
* вызывает Renderer
* возвращает результат

Протокол взаимодействия определяется на позднем этапе.

== UI (Qt / QML)

UI является клиентом движка и не содержит бизнес-логики.

Назначение:

* управление блоками
* сборка композиций
* публикация версий
* предпросмотр результата

== Roadmap

=== Этап 0 — Proof of Concept

Поддерживается:

* Block
* Composition
* Renderer
* plain text output

=== Этап 1 — MVP

Поддерживается:

* параметры
* версии Block
* сериализация
* publishing

=== Этап 2 — Stabilization

Поддерживается:

* StyleProfile
* Normalizer API
* diff версий
* расширенный поиск

=== Этап 3 — Interfaces

* Prompt Server
* UI
* CLI / bindings

== Non-goals

* автоматическая генерация текста
* скрытое использование LLM
* UI как источник бизнес-логики
* Prompt Server как редактор
* универсальный формат данных

== Критерии успешности

Проект считается успешным, если:

* движок используется ежедневно
* повторяющиеся тексты исчезают
* изменение структуры не требует переписывания контента

=== Layer Structure

=== Core API (Public Interface)

[source,cpp]
----
namespace textengine {

class Engine {
public:
    explicit Engine(std::unique_ptr<IBlockRepository> blocks,
                   std::unique_ptr<ICompositionRepository> compositions);

    // Block Lifecycle
    Block create_block_draft(BlockId id, Template tmpl, Params defaults);
    Block publish_block(const BlockId& id);  // Draft -> Published (immutable)
    Block deprecate_block(const BlockId& id, Version ver);  // Published -> Deprecated

    // Composition Lifecycle
    Composition create_composition_draft(CompositionId id);
    Composition publish_composition(const CompositionId& id);

    // Rendering
    std::string render(const CompositionId& id, const RenderContext& ctx);
};

} // namespace textengine
----

== Parameter Resolution Strategy

Параметры разрешаются по иерархии (от менее приоритетных к более приоритетным):

1. **Block Defaults** — параметры, определённые в Draft/Published Block
2. **Composition Local Override** — `local_params` в `BlockRef` внутри Composition
3. **Runtime Context** — параметры, переданные в `RenderContext` при вызове `render()`

[source,cpp]
----
// Пример разрешения параметров:
Block "role.writer" {
    defaults: { "tone": "нейтральный", "language": "ru" }
}

Composition "blog.post" {
    block_refs: [{
        block_id: "role.writer",
        local_params: { "tone": "энтузиастский" }  // override #1
    }]
}

// Runtime:
RenderContext ctx;
ctx.params = { "language": "en" };  // override #2

// Result: tone="энтузиастский", language="en"
----

=== Strict vs Default Parameters

**Вариант A (Default):** Block содержит `default_params`. Если параметр не переопределён на уровне Composition или Runtime, используется default.

**Вариант B (Strict):** Block содержит `params_schema` с флагом `required`. Отсутствие значения на этапе render вызывает ошибку (соответствует принципу "нет магии").

== Storage Schema (ObjectBox)

=== Entities

[source,cpp]
----
// BlockEntity — хранит версии блоков
struct BlockEntity {
    obx_id obx_id;                    // internal ObjectBox ID
    std::string block_id;             // user-facing ID (e.g., "role.system")
    std::string project_key;          // namespace isolation

    int state;                        // 0=Draft, 1=Published, 2=Deprecated
    int64_t version_major;            // Semantic versioning
    int64_t version_minor;

    std::string template_content;     // Raw template with {{params}}
    std::string default_params_json;  // Default values (for Draft/Published)
    std::string params_schema_json;   // Validation schema (optional)
    std::vector<std::string> tags;
    int64_t created_at;               // Unix timestamp

    // Critical Constraint: Published entities are immutable
    bool is_mutable() const { return state == 0; }
};

// CompositionEntity — хранит структуру композиции
struct CompositionEntity {
    obx_id obx_id;
    std::string composition_id;
    std::string project_key;
    int state;

    // JSON array of BlockRef:
    // [
    //   {"block_id": "role.system", "version": "1.2",
    //    "local_params": {"tone": "formal"}},
    //   {"block_id": "format.json", "version": "2.0", "local_params": {}}
    // ]
    std::string block_refs_json;

    std::string target_language;
    std::string style_profile_id;
    int64_t created_at;
};

== Workflow Example (Library Usage)

Полный пример использования движка без CLI:

[source,cpp]
----
// 1. Initialize storage
auto store = std::make_unique<ObjectBoxStore>("./project.obx");
textengine::Engine engine(store->block_repo(), store->comp_repo());

// 2. Create Block Draft with defaults
auto role_draft = Block::create_draft("role.tech-writer");
role_draft.set_template("Ты пишешь для {{audience}}. Стиль: {{tone}}.");
role_draft.set_default_params({
    {"audience", "разработчиков"},
    {"tone", "нейтральный"}
});

auto role_block = engine.create_block_draft(role_draft);

// 3. Publish Block (make immutable, assign version 1.0)
auto role_published = engine.publish_block(role_block.id());

// 4. Create Composition
auto comp_draft = Composition::create_draft("blog.golang");
comp_draft.add_block_ref({
    .block_id = role_published.id(),
    .version = role_published.version(),
    .local_params = {{"tone", "энтузиастский"}}  // Override default
});

auto comp = engine.create_composition_draft(comp_draft);
auto comp_published = engine.publish_composition(comp.id());

// 5. Render (Runtime parameter override)
RenderContext ctx;
ctx.params = {{"audience", "senior-разработчиков"}};  // Override again

std::string final_text = engine.render(comp_published.id(), ctx);
// Result: "Ты пишешь для senior-разработчиков. Стиль: энтузиастский."
----

== Critical Constraints (Implementation)

1. **Draft Parameters**: Draft может содержать `default_params`. Они становятся неизменяемыми при переходе в Published.

2. **Immutable Published**: После `publish_block()` сущность записывается в storage с флагом Published и становится read-only. Любое изменение создаёт новую версию через Draft.

3. **Composition BlockRef**: При добавлении блока в Composition обязательно указание версии (`version`). Это обеспечивает воспроизводимость: даже если выйдет Block v1.1, Composition продолжит использовать v1.0 пока не обновят вручную.

4. **Rendering Failure**: Если параметр отсутствует в defaults, local_params и runtime context — render возвращает `Error::MissingParam` вместо магической подстановки пустой строки (Design Constraint: контроль важнее удобства).

== Prompt Server Access Model

Prompt Server работает только с Published сущностями:

[source]
----
Client Request: GET /prompts/blog.golang?v=1.0&audience=junior
                ↓
Prompt Server: engine.render("blog.golang", v1.0, {audience: "junior"})
                ↓
Response: {
  "text": "Ты пишешь для junior...",
  "meta": {
    "composition_version": "1.0",
    "blocks": [
      {"id": "role.tech-writer", "version": "1.0"},
      {"id": "format.article", "version": "2.1"}
    ]
  }
}
----

Prompt Server никогда не вызывает `publish_*` или `save_draft` методы (Read-only режим из PRD).
