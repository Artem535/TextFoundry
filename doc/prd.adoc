== Цель документа

Этот документ описывает ограничения, архитектурные принципы, жизненные циклы сущностей и roadmap *Text Engine* — движка для конструирования текстов и промтов из переиспользуемых смысловых блоков.

Документ служит:

* фиксацией инженерных решений
* защитой от расползания требований
* контрактом между Core, Storage, Server и UI
* ориентиром для будущей реализации (CLI / GUI / интеграции)

== Область применения

Text Engine предназначен для:

* сборки текстов из логических блоков
* переиспользования фрагментов между разными текстами
* параметризации и вариативности формулировок
* контролируемой нормализации стиля

Text Engine *не является*:

* LLM-фреймворком
* системой автоматического перевода
* WYSIWYG-редактором

== Ключевые ограничения (Design Constraints)

=== Принцип engine-first

Движок:

* разрабатывается как библиотека
* не зависит от UI, CLI или сетевого протокола
* имеет единый публичный API

Любые интерфейсы считаются *клиентами* движка.

=== Отсутствие "магии"

На ранних этапах движок:

* не выполняет семантическое переписывание текста
* не меняет язык или время автоматически
* не пытается угадывать намерения пользователя

Все преобразования должны быть:

* детерминированными
* воспроизводимыми
* тестируемыми

=== Контроль над текстом важнее удобства

Если автоматизация приводит к:

* потере смысла
* размытию формулировок
* скрытым изменениям текста

— такая автоматизация запрещена.

== Основные сущности

=== Block

Логический переиспользуемый фрагмент текста.

Характеристики:

* уникальный идентификатор
* тип (role, constraint, style, domain, meta)
* шаблон содержания
* параметры
* теги
* язык
* версия
* состояние жизненного цикла

Block не знает:

* где он используется
* в каком порядке
* в каком финальном тексте

=== Composition

Композиция блоков, образующая конкретный текст.

Свойства:

* упорядоченный список Block (с указанием версий)
* локальные параметры
* целевой язык
* целевой стиль
* состояние жизненного цикла

Composition не содержит текст напрямую — только структуру.

=== StyleProfile

Описание требований к финальному тексту.

Может включать:

* язык
* тон
* время (tense)
* форматирование

StyleProfile задаёт *намерение*, но не гарантирует автоматическое приведение текста.

=== Renderer

Компонент, выполняющий:

* подстановку параметров
* разворачивание шаблонов
* сборку итогового текста

Renderer не выполняет:

* перевод
* переформулирование
* смысловую коррекцию

== Жизненный цикл сущностей

Все основные сущности (Block, Composition) имеют явный жизненный цикл.

=== Состояния

* *Draft* — редактируемая версия
* *Published* — зафиксированная неизменяемая версия
* *Deprecated* — устаревшая, но доступная для чтения

=== Правила

* Draft можно изменять без ограничений
* Published версии:
** неизменяемы
** адресуются по версии
** могут использоваться Prompt Server
* Любое изменение Published сущности создаёт новую версию
* Deprecated версии:
** не удаляются
** не используются по умолчанию

== Этапы обработки текста

[source]
----
Blocks
↓
Template Expansion
↓
Raw Text
↓
(optional) Normalization
↓
Final Text
----

=== Template Expansion

Детерминированная стадия:

* подстановка параметров
* разворачивание словарей
* выбор вариантов

=== Normalization (опционально)

Отдельный этап, который:

* может отсутствовать
* может быть реализован через LLM
* может быть отключён полностью

Нормализация никогда не является обязательной.

== Поддержка языков и времени

=== Базовая стратегия

На уровне движка поддерживаются только:

* словари терминов
* шаблонные конструкции
* нейтральные формулировки

=== Что НЕ поддерживается без LLM

* полноценный перевод текста
* автоматическое изменение времени
* сохранение стиля при переписывании

== Storage Model

Storage является инфраструктурным слоем и полностью отделён от бизнес-логики движка.

=== Принципы хранения

* движок не зависит от конкретной БД
* работа идёт только через репозитории
* формат хранения не является частью публичного API

=== Проекты и пространства имён

Минимальная адресация:

* project_key
* entity_type
* entity_id
* version

Разделение по проектам является обязательным требованием для Prompt Server.

=== Read / Write модели

Допускается разделение режимов:

* read-write — UI, редакторы
* read-only — Prompt Server

Prompt Server никогда не работает с Draft-данными.

=== Версионирование

Storage обязан поддерживать:

* хранение версий
* доступ к конкретной версии
* неизменяемость опубликованных сущностей

== Publishing и синхронизация

=== Publishing

Publishing — явная операция, которая:

* фиксирует текущую Draft-версию
* создаёт Published-версию
* делает её доступной для Prompt Server

Publishing является необратимым действием.

=== Sync модель

* UI работает с Workspace (Draft)
* Prompt Server читает только Published Storage
* Обновление промта = публикация новой версии
* Старые версии продолжают обслуживаться

'''

== 🔧 CLI и Terminal UI (обязательный интерфейс)

=== CLI (обязательный компонент)

CLI является **обязательным интерфейсом** системы и рассматривается как *reference client* для Text Engine.

CLI предназначен для:

* проверки полноты и удобства Core API
* инженерного использования без GUI
* автоматизации (скрипты, CI, dev workflows)
* публикации и инспекции промтов

CLI — это **не временный инструмент**, а архитектурный якорь проекта.

==== Роль CLI в архитектуре

CLI:

* является каноническим способом взаимодействия с Core API
* использует те же абстракции, что и Prompt Server / UI
* не содержит бизнес-логики
* не обходит ограничения движка

Любая функциональность, добавляемая в Core, **должна быть достижима через CLI**.

Если операция невозможна через CLI — это архитектурная проблема.

==== Назначение CLI

CLI используется для:

* управления блоками
* управления композициями
* рендеринга текста
* работы с версиями
* публикации промтов
* диагностики и инспекции структуры

CLI **не предназначен** для:

* визуального редактирования сложных структур
* WYSIWYG-работы
* замены GUI

==== Примерные сценарии использования

[source]
----
tf block list
tf block show role.system
tf block render role.system --lang=en

tf composition inspect finance_advisor
tf composition render finance_advisor --version=1.2

tf prompt publish finance_advisor
tf prompt diff finance_advisor --from=1.1 --to=1.2
----

Командный интерфейс считается стабильным API для инженеров.

'''

=== Terminal UI (TUI) — допустимое расширение

Terminal UI (TUI) допускается как **надстройка над CLI**, но не как отдельный интерфейс с собственной логикой.

TUI может быть реализован с использованием:

* ftxui
* termox2

Выбор конкретной библиотеки не фиксируется в PRD.

==== Назначение TUI

TUI предназначен для:

* удобной навигации по структуре блоков
* просмотра композиций
* инспекции зависимостей
* быстрого предпросмотра рендеринга

TUI **не заменяет**:

* Qt/QML UI
* Core API
* CLI

==== Архитектурные ограничения TUI

TUI:

* использует CLI или Core API напрямую
* не содержит логики рендеринга
* не управляет версиями напрямую
* не имеет собственного слоя хранения

TUI рассматривается как *UX-обёртка* над CLI, а не как самостоятельный клиент.

==== Статус реализации

TUI:

* не является обязательным для MVP
* может быть реализован после стабилизации CLI
* не влияет на критерии успешности проекта

Отсутствие TUI **не блокирует** развитие Prompt Server или UI.

== Prompt Server

=== Назначение

Prompt Server — сервис дистрибуции и разрешения промтов.

Он:

* не редактирует данные
* не генерирует новые версии
* не содержит логики рендеринга

=== Запрос клиента

Клиент передаёт:

* project_key
* prompt_name (или путь)
* параметры (опционально)

=== Ответ сервера

Сервер возвращает:

* финальный текст
* метаданные:
** версия
** язык
** стиль

=== Внутренняя логика

Prompt Server:

* разрешает Composition по имени
* загружает зафиксированные версии Block
* вызывает Renderer
* возвращает результат

Протокол взаимодействия определяется на позднем этапе.

== UI (Qt / QML)

UI является клиентом движка и не содержит бизнес-логики.

Назначение:

* управление блоками
* сборка композиций
* публикация версий
* предпросмотр результата

== Roadmap

=== Этап 0 — Proof of Concept

Поддерживается:

* Block
* Composition
* Renderer
* plain text output

=== Этап 1 — MVP

Поддерживается:

* параметры
* версии Block
* сериализация
* publishing

=== Этап 2 — Stabilization

Поддерживается:

* StyleProfile
* Normalizer API
* diff версий
* расширенный поиск

=== Этап 3 — Interfaces

* Prompt Server
* UI
* CLI / bindings

== Non-goals

* автоматическая генерация текста
* скрытое использование LLM
* UI как источник бизнес-логики
* Prompt Server как редактор
* универсальный формат данных

== Критерии успешности

Проект считается успешным, если:

* движок используется ежедневно
* повторяющиеся тексты исчезают
* изменение структуры не требует переписывания контента
