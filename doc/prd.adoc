== Цель документа

Этот документ описывает ограничения, архитектурные принципы, этапы разработки и roadmap *Text Engine* — движка для конструирования текстов и промтов из переиспользуемых смысловых блоков.

Документ служит:

* фиксацией инженерных решений
* защитой от расползания требований
* ориентиром для будущей разработки (CLI / GUI / интеграции)

== Область применения

Text Engine предназначен для:

* сборки текстов из логических блоков
* переиспользования фрагментов между разными текстами
* параметризации и вариативности формулировок
* контролируемой нормализации стиля

Text Engine *не является*:

* LLM-фреймворком
* системой автоматического перевода
* WYSIWYG-редактором

== Ключевые ограничения (Design Constraints)

=== Принцип engine-first

Движок:

* разрабатывается как библиотека
* не зависит от UI, CLI или LLM
* имеет один публичный API

Любые интерфейсы (CLI, GUI, плагины) считаются *клиентами* движка.

=== Отсутствие "магии"

На ранних этапах движок:

* не выполняет семантическое переписывание текста
* не меняет язык или время автоматически
* не пытается "угадывать намерения"

Все преобразования должны быть:

* детерминированными
* воспроизводимыми
* тестируемыми

=== Контроль над текстом важнее удобства

Если автоматизация приводит к:

* потере смысла
* размытию формулировок
* скрытым изменениям текста

— такая автоматизация запрещена.

== Основные сущности

=== Block

Логический переиспользуемый фрагмент текста.

Характеристики:

* уникальный идентификатор
* тип (role, constraint, style, domain, meta)
* шаблон содержания
* параметры
* теги
* язык
* версия

Block не знает:

* где он используется
* в каком порядке
* в каком финальном тексте

=== Composition

Композиция блоков, образующая конкретный текст.

Свойства:

* упорядоченный список Block
* локальные параметры
* целевой язык
* целевой стиль

Composition не содержит текст напрямую — только структуру.

=== StyleProfile

Описание требований к финальному тексту.

Может включать:

* язык
* тон
* время (tense)
* форматирование

StyleProfile *не гарантирует* автоматическое приведение текста — он задаёт намерение.

=== Renderer

Компонент, выполняющий:

* подстановку параметров
* разворачивание шаблонов
* сборку итогового текста

Renderer не выполняет:

* перевод
* переформулирование
* смысловую коррекцию

== Этапы обработки текста

[source]
----
Blocks
↓
Template Expansion
↓
Raw Text
↓
(optional) Normalization
↓
Final Text
----

=== Template Expansion

Детерминированная стадия:

* подстановка параметров
* разворачивание словарей
* выбор вариантов

=== Normalization (опционально)

Отдельный этап, который:

* может отсутствовать
* может быть реализован через LLM
* может быть отключён полностью

Нормализация никогда не является обязательной.

== Поддержка языков и времени

=== Базовая стратегия

На уровне движка поддерживаются только:

* словари терминов
* шаблонные конструкции
* нейтральные формулировки

=== Что НЕ поддерживается без LLM

* полноценный перевод текста
* автоматическое изменение времени
* сохранение стиля при переписывании

Любые попытки реализовать это правилами считаются экспериментальными.

== Storage Model

Данный раздел описывает модель хранения данных и границы ответственности слоя хранения.

Storage рассматривается как *инфраструктурный слой*, полностью отделённый от бизнес-логики Text Engine.

=== Принципы хранения

* Text Engine не зависит от конкретной реализации БД
* Движок работает только через абстракции репозиториев
* Storage может быть локальным или удалённым
* Формат хранения не является частью публичного API

=== Разделение ответственности

* Core-модель (Block, Composition, StyleProfile) не содержит логики хранения
* Storage отвечает только за:
  ** сохранение сущностей
  ** загрузку сущностей
  ** версионирование
  ** поиск и фильтрацию

=== Проекты и пространства имён

Хранилище поддерживает логическое разделение по проектам.

Минимальная адресация:

* project_key
* entity_type
* entity_name / id

Это разделение является обязательным требованием для Prompt Server.

=== Read / Write модели

Допускается разделение режимов доступа:

* read-write (UI, редакторы)
* read-only (Prompt Server)

Prompt Server может использовать:

* ту же БД в read-only режиме
* реплику
* кэшированное представление

=== Версионирование

Storage обязан поддерживать:

* версии Block
* возможность получения конкретной версии
* неизменяемость опубликованных версий

Политика версионирования определяется на уровне движка.

=== Реализация по умолчанию

На этапе MVP используется:

* локальное хранилище
* ObjectBox

ObjectBox рассматривается как *референсная реализация*, а не как жёсткая зависимость.

=== Будущие реализации (необязательно)

Возможные альтернативы:

* файловое хранилище (JSON / YAML)
* embedded SQL
* сетевое хранилище

Поддержка альтернативных реализаций не является обязательной для успешности проекта.

== Хранилище данных

На этапе MVP:

* локальное хранилище
* ObjectBox

Причины выбора:

* граф связей
* версии сущностей
* высокая производительность

== Будущая архитектура и расширения

Данный раздел фиксирует стратегические направления развития системы, которые *не реализуются на ранних этапах*, но являются обязательными целями проекта.

=== Prompt Server (обязательное расширение)

Одной из ключевых долгосрочных целей проекта является *Prompt Server* — сервис для централизованного хранения и выдачи промтов клиентским приложениям.

==== Назначение Prompt Server

Prompt Server предназначен для:

* получения готовых промтов по ключу проекта
* адресации промтов по имени (пути) файла
* обеспечения единого источника истины для промтов
* использования движка Text Engine как backend-ядра

Сервер решает задачу *дистрибуции и версионирования промтов*, а не их генерации.

==== Модель доступа

Клиент делает запрос вида:

* project_key
* prompt_name (или путь)
* опциональные параметры

В ответ сервер возвращает:

* собранный текст промта
* метаданные (версия, язык, стиль)

==== Архитектурные требования

Prompt Server:

* использует Text Engine как библиотеку
* не содержит бизнес-логики рендеринга
* не знает о UI
* может работать как локально, так и в сетевом режиме

Протокол взаимодействия (HTTP / gRPC / IPC) определяется на позднем этапе.

==== Версии и стабильность

Prompt Server должен обеспечивать:

* стабильные ключи доступа
* возможность изменения внутренней структуры промта без изменения API
* совместимость клиентов между версиями

=== UI (Qt / QML)

UI на базе Qt/QML с backend на C++ рассматривается как *основной пользовательский интерфейс* для работы с системой.

==== Назначение UI

UI предназначен для:

* управления блоками
* сборки композиций
* визуального контроля структуры текста
* предпросмотра финального результата

UI *не является* обязательным компонентом для работы Prompt Server.

==== Архитектурные принципы UI

* UI — клиент движка
* вся логика находится в C++ backend
* QML используется только для представления и взаимодействия

=== Приоритеты развития

Порядок приоритетов фиксирован:

1. Проектирование и стабилизация Text Engine
2. Prompt Server
3. UI (Qt/QML)
4. CLI и прочие интерфейсы

Изменение этого порядка требует осознанного пересмотра архитектуры.

== Roadmap

=== Этап 0 — Proof of Concept

Цель: доказать полезность подхода.

Поддерживается:

* Block
* Composition
* Renderer
* plain text output

Не поддерживается:

* версии
* стили
* языки
* LLM

Оценка: 2–4 дня

=== Этап 1 — MVP

Поддерживается:

* параметры блоков
* словари терминов
* версии Block
* сериализация

Ограничения:

* ручной контроль языка
* отсутствие авто-нормализации

Оценка: 1–2 недели

=== Этап 2 — Stabilization

Поддерживается:

* StyleProfile
* Normalizer API
* diff версий
* расширенный поиск

LLM:

* опционально
* как плагин

Оценка: 2–3 недели

=== Этап 3 — Interfaces

Возможные клиенты:

* CLI
* Qt/QML GUI
* Python bindings

Решение о разработке принимается *после активного использования движка*.

== Осознанные ограничения

Text Engine сознательно:

* не решает задачу NLG
* не конкурирует с переводчиками
* не скрывает сложность языка

Это инструмент для инженера, а не для автоматической генерации текстов.

== Non-goals (Осознанно не поддерживается)

Данный раздел фиксирует задачи и направления, которые *намеренно исключены* из области ответственности проекта.

Этот список предназначен для предотвращения расползания требований и пересмотру не подлежит без серьёзных архитектурных причин.

=== Автоматическая генерация текста

Text Engine:

* не является системой NLG
* не генерирует новый контент
* не дописывает текст за пользователя

Движок работает только с явно заданными блоками.

=== Полноценный перевод и переписывание

Проект:

* не реализует собственный переводчик
* не гарантирует корректное изменение языка
* не выполняет автоматическое переписывание формулировок

Любые такие операции возможны только через опциональные внешние нормализаторы.

=== Скрытое использование LLM

LLM:

* никогда не используется неявно
* не является обязательной зависимостью
* не встраивается внутрь Renderer

Все вызовы LLM должны быть:

* явными
* конфигурируемыми
* отключаемыми

=== UI как источник бизнес-логики

UI:

* не содержит логики рендеринга
* не управляет версионированием
* не принимает архитектурные решения

UI является клиентом движка, а не его частью.

=== Prompt Server как редактор

Prompt Server:

* не предназначен для редактирования блоков
* не изменяет структуру композиций
* не создаёт новые версии

Сервер предоставляет только доступ к уже опубликованным промтам.

=== Универсальный формат данных

Проект:

* не стремится создать универсальный формат текстов
* не обеспечивает совместимость со сторонними редакторами
* не стандартизирует язык описания блоков

Внутренние форматы могут свободно изменяться.

=== Максимальная автоматизация

Проект сознательно:

* не оптимизируется под "одну кнопку"
* не скрывает сложность языка
* не жертвует контролем ради удобства

== Критерии успешности

Проект считается успешным, если:

* разработчик использует движок ежедневно
* повторяющиеся тексты исчезают
* изменения структуры не требуют переписывания контента

Если движок не используется — он признаётся избыточным и замораживается.
