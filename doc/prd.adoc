== Цель и назначение документа [[purpose]]

Этот документ описывает ограничения, архитектурные принципы, жизненные циклы сущностей и дорожную карту *Text Engine* — движка для конструирования текстов и промтов из переиспользуемых смысловых блоков [1].

Документ служит:
* фиксацией инженерных решений
* защитой от расползания требований
* контрактом между Core, Storage, Server и UI
* ориентиром для будущей реализации (CLI / GUI / интеграции)

=== Область применения [[scope]]

Text Engine предназначен для:

* сборки текстов из логических блоков
* переиспользования фрагментов между разными текстами
* параметризации и вариативности формулировок
* контролируемой нормализации стиля

Text Engine *не является*:

* LLM-фреймворком
* системой автоматического перевода
* WYSIWYG-редактором

== Ключевые ограничения (Design Constraints) [[design-constraints]]

=== Принцип engine-first [[engine-first]]
Движок разрабатывается как самостоятельная библиотека [1], которая:
* не зависит от UI, CLI или сетевого протокола
* имеет единый публичный API
* рассматривает любые интерфейсы как *клиентов*

=== Отсутствие "магии" [[no-magic]]
На ранних этапах запрещены операции, нарушающие детерминизм [1]:
* семантическое переписывание текста
* автоматическое изменение языка или времени
* попытки угадать намерения пользователя

Все преобразования должны быть детерминированными, воспроизводимыми и тестируемыми [1].

=== Контроль над текстом важнее удобства [[control-over-convenience]]
Если автоматизация приводит к потере смысла, размытию формулировок или скрытым изменениям текста [1] — такая автоматизация запрещена.

== Основные сущности [[entities]]

=== Блок (Block) [[block]]
Логический переиспользуемый фрагмент текста с характеристиками [1]:

* уникальный идентификатор
* тип (role, constraint, style, domain, meta)
* шаблон содержания, параметры, теги
* язык, версия и состояние жизненного цикла

Block не знает [1]:

* где он используется
* в каком порядке
* в каком финальном тексте

=== Композиция (Composition) [[composition]]

Композиция блоков, образующая конкретный текст [1]:

* упорядоченный список Block (с указанием версий)
* локальные параметры
* целевой язык и стиль
* состояние жизненного цикла

Composition не содержит текст напрямую — только структуру [1].

=== Профиль стиля (StyleProfile) [[style-profile]]

Описание требований к финальному тексту [1]:

* язык, тон, время (tense), форматирование

StyleProfile задаёт *намерение*, но не гарантирует автоматическое приведение текста [1].

=== Рендерер (Renderer) [[renderer]]
Компонент, выполняющий подстановку параметров, разворачивание шаблонов и сборку итогового текста [1].

Renderer не выполняет перевод, переформулирование или смысловую коррекцию [1].

== Жизненный цикл сущностей [[lifecycle]]

=== Состояния [[lifecycle-states]]
* *Draft* — редактируемая версия [1]
* *Published* — зафиксированная неизменяемая версия [1]
* *Deprecated* — устаревшая, но доступная для чтения [1]

=== Правила переходов [[lifecycle-rules]]
* Draft можно изменять без ограничений [1]
* Published версии неизменяемы, адресуются по версии и могут использоваться <<prompt-server,Prompt Server>> [1]
* Любое изменение Published сущности создаёт новую версию через Draft [1]
* Deprecated версии не удаляются и не используются по умолчанию [1]

== Этапы обработки текста [[text-processing]]

[source]
----
Blocks
↓
Template Expansion
↓
Raw Text
↓
(optional) Normalization
↓
Final Text
----

=== Разворачивание шаблонов (Template Expansion) [[template-expansion]]
Детерминированная стадия [1]:

* подстановка параметров
* разворачивание словарей
* выбор вариантов

=== Нормализация (Normalization) [[normalization]]
Опциональный этап [1]:

* может отсутствовать
* может быть реализован через LLM
* может быть отключён полностью

Нормализация никогда не является обязательной [1].

== Поддержка языков и времени (i18n) [[i18n]]

=== Базовая стратегия [[i18n-strategy]]
На уровне движка поддерживаются только [1]:

* словари терминов
* шаблонные конструкции
* нейтральные формулировки

=== Ограничения без LLM [[i18n-limitations]]
Следующие функции не поддерживаются без привлечения LLM [1]:

* полноценный перевод текста
* автоматическое изменение времени
* сохранение стиля при переписывании

== Модель хранения (Storage Model) [[storage-model]]

Storage является инфраструктурным слоем и полностью отделён от бизнес-логики движка [1].

=== Принципы хранения [[storage-principles]]

* движок не зависит от конкретной БД [1]
* работа идёт только через репозитории [1]
* формат хранения не является частью публичного API [1]

=== Проекты и пространства имён [[namespaces]]
Минимальная адресация сущностей [1]:

* `project_key`
* `entity_type`
* `entity_id`
* `version`

Разделение по проектам является обязательным требованием для <<prompt-server,Prompt Server>> [1].

=== Режимы доступа (Read/Write) [[access-modes]]
Допускается разделение режимов [1]:

* **read-write** — используется UI и редакторами
* **read-only** — используется Prompt Server

Важно: Prompt Server никогда не работает с Draft-данными [1].

=== Версионирование [[versioning]]
Storage обязан поддерживать [1]:

* хранение версий и доступ к конкретной версии
* неизменяемость опубликованных сущностей

== Публикация и синхронизация [[publishing]]

=== Операция публикации [[publishing-operation]]
Publishing — явная необратимая операция [1]:

* фиксирует текущую Draft-версию
* создаёт Published-версию
* делает её доступной для <<prompt-server,Prompt Server>>

=== Модель синхронизации [[sync-model]]

* UI работает с Workspace (Draft) [1]
* Prompt Server читает только Published Storage [1]
* Обновление промта = публикация новой версии [1]
* Старые версии продолжают обслуживаться [1]

== Интерфейсы: CLI и Terminal UI [[interfaces]]

=== CLI (обязательный компонент) [[cli]]
CLI является *reference client* для Text Engine и архитектурным якорем проекта [1].

Назначение CLI [1]:

* проверка полноты Core API
* инженерное использование без GUI
* автоматизация (скрипты, CI)
* публикация и инспекция промтов

CLI не предназначен для визуального редактирования или замены GUI [1].

=== Terminal UI (TUI) [[tui]]
TUI рассматривается как *UX-обёртка* над CLI, а не как самостоятельный клиент [1].

Архитектурные ограничения TUI [1]:

* использует CLI или Core API напрямую
* не содержит логики рендеринга
* не управляет версиями напрямую
* не имеет собственного слоя хранения

Статус реализации: не является обязательным для MVP, может быть реализован после стабилизации CLI [1]. Отсутствие TUI не блокирует развитие <<prompt-server,Prompt Server>> или UI [1].

== Сервер промптов (Prompt Server) [[prompt-server]]

=== Назначение и ограничения [[prompt-server-purpose]]
Prompt Server — сервис дистрибуции и разрешения промтов [1]:

* не редактирует данные
* не генерирует новые версии
* не содержит логики рендеринга (делегирует <<renderer,Renderer>>)

=== Протокол взаимодействия [[prompt-server-protocol]]
Запрос клиента [1]:

* `project_key`
* `prompt_name` (или путь)
* параметры (опционально)

Ответ сервера [1]:

* финальный текст
* метаданные (версия, язык, стиль)

Внутренняя логика: разрешает <<composition,Composition>>, загружает зафиксированные версии <<block,Block>> через <<storage-model,Storage>> (только <<access-modes,read-only>> режим), вызывает <<renderer,Renderer>> [1].

== UI (Qt/QML) [[ui]]
UI является клиентом движка и не содержит бизнес-логики [1].

Назначение:

* управление блоками и сборка <<composition,Composition>>
* публикация версий
* предпросмотр результата

== Дорожная карта (Roadmap) [[roadmap]]

=== Этап 0 — Proof of Concept [[roadmap-poc]]
* Block, Composition, <<renderer,Renderer>>
* plain text output

=== Этап 1 — MVP [[roadmap-mvp]]
* параметры и версии Block
* сериализация и <<publishing,publishing>>

=== Этап 2 — Stabilization [[roadmap-stabilization]]
* <<style-profile,StyleProfile>>
* Normalizer API
* diff версий и расширенный поиск

=== Этап 3 — Interfaces [[roadmap-interfaces]]
* <<prompt-server,Prompt Server>>
* <<ui,UI>>
* <<cli,CLI>> / bindings

== Границы проекта (Non-goals) [[non-goals]]
* автоматическая генерация текста [1]
* скрытое использование LLM [1]
* UI как источник бизнес-логики [1]
* Prompt Server как редактор [1]
* универсальный формат данных [1]

== Критерии успешности [[success-criteria]]
Проект считается успешным, если [1]:

* движок используется ежедневно
* повторяющиеся тексты исчезают
* изменение структуры не требует переписывания контента

== Архитектура реализации (C++ Core) [[implementation]]

=== Структура слоёв [[layer-structure]]
[source]
----
┌─────────────────────────────────────┐
│  CLI / TUI / Qt GUI / Prompt Server │  <- Clients (reference: CLI)
├─────────────────────────────────────┤
│  TextEngine Core (header-only API)  │  <- Pure C++17, zero UI deps
│  - Engine                           │
│  - Renderer                         │
│  - Block/Composition domain models  │
├─────────────────────────────────────┤
│  Repository Abstraction             │  <- Ports (interfaces)
│  IBlockRepository                   │
│  ICompositionRepository             │
├─────────────────────────────────────┤
│  ObjectBox Storage Layer            │  <- Adapter implementation
│  - BlockEntity                      │
│  - CompositionEntity                │
│  - Version indexing                 │
└─────────────────────────────────────┘
----

=== Стратегия разрешения параметров [[parameter-resolution]]
Параметры разрешаются по иерархии (от менее приоритетных к более приоритетным) [1]:

1. **Block Defaults** — параметры, определённые в Draft/Published <<block,Block>>
2. **Composition Local Override** — `local_params` в `BlockRef` внутри <<composition,Composition>>
3. **Runtime Context** — параметры, переданные в `RenderContext` при вызове `render()`

=== Строгие vs опциональные параметры [[strict-vs-default]]
* **Default**: Block содержит `default_params`. Если параметр не переопределён, используется значение по умолчанию [1].
* **Strict**: Block содержит `params_schema` с флагом `required`. Отсутствие значения вызывает ошибку (соответствует принципу <<no-magic,"нет магии">>).

== Схема хранения (ObjectBox) [[storage-schema]]

=== Сущности [[storage-entities]]
*BlockEntity* — хранит версии блоков с полями: `block_id`, `project_key`, `state` (0=Draft, 1=Published, 2=Deprecated), версионирование, шаблон, параметры [1].

*CompositionEntity* — хранит структуру композиции: `composition_id`, `project_key`, `block_refs_json` (массив ссылок на блоки с версиями), целевой язык и стиль [1].

=== Паттерн Repository [[repository-pattern]]
`IBlockRepository` и `ICompositionRepository` — чистые интерфейсы (ports), не зависящие от ObjectBox [1]. Реализация `ObjectBoxBlockRepository` содержит специфику хранилища [1].

== Пример использования (Library Usage) [[workflow-example]]
Полный цикл: инициализация хранилища, создание Draft <<block,Block>> с параметрами по умолчанию, <<publishing-operation,публикация>>, создание <<composition,Composition>> с переопределением параметров, рендеринг с Runtime-контекстом [1].

== Критические ограничения реализации [[critical-constraints]]

. **Draft Parameters**: становятся неизменяемыми при переходе в Published [1]
. **Immutable Published**: сущность становится read-only, изменение создаёт новую версию [1]
. **Composition BlockRef**: обязательное указание версии (`version`) для воспроизводимости [1]
. **Rendering Failure**: отсутствие параметра возвращает `Error::MissingParam` вместо магической подстановки (соответствует <<control-over-convenience,контролю над текстом>>) [1]

== Модель доступа Prompt Server [[prompt-server-access]]
Prompt Server работает только с Published сущностями [1]:

* Запрос: `GET /prompts/blog.golang?v=1.0&audience=junior`
* Обработка: `engine.render("blog.golang", v1.0, {audience: "junior"})`
* Ответ: финальный текст + метаданные использованных блоков

Prompt Server никогда не вызывает `publish_*` или `save_draft` методы (<<access-modes,read-only режим>>) [1].
