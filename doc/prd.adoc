== Цель и назначение документа [[purpose]]

Этот документ описывает ограничения, архитектурные принципы, жизненные циклы сущностей и дорожную карту *Text Engine* — движка для конструирования текстов и промтов из переиспользуемых смысловых блоков [1].

Документ служит:

* фиксацией инженерных решений
* защитой от расползания требований
* контрактом между Core, Storage, Server и UI
* ориентиром для будущей реализации (CLI / GUI / интеграции)

=== Область применения [[scope]]

Text Engine предназначен для:

* сборки текстов из логических блоков
* переиспользования фрагментов между разными текстами
* параметризации и вариативности формулировок
* контролируемой нормализации стиля

Text Engine *не является*:

* LLM-фреймворком
* системой автоматического перевода
* WYSIWYG-редактором

== Ключевые ограничения (Design Constraints) [[design-constraints]]

=== Принцип engine-first [[engine-first]]
Движок разрабатывается как самостоятельная библиотека [1], которая:

* не зависит от UI, CLI или сетевого протокола
* имеет единый публичный API
* рассматривает любые интерфейсы как *клиентов*

=== Отсутствие "магии" [[no-magic]]
На ранних этапах запрещены операции, нарушающие детерминизм [1]:

* семантическое переписывание текста
* автоматическое изменение языка или времени
* попытки угадать намерения пользователя

Все преобразования должны быть детерминированными, воспроизводимыми и тестируемыми [1].

=== Контроль над текстом важнее удобства [[control-over-convenience]]
Если автоматизация приводит к потере смысла, размытию формулировок или скрытым изменениям текста [1] — такая автоматизация запрещена.

== Основные сущности [[entities]]

=== Блок (Block) [[block]]
Логический переиспользуемый фрагмент текста с характеристиками [1]:

* уникальный идентификатор
* тип (role, constraint, style, domain, meta)
* шаблон содержания, параметры, теги
* язык, версия и состояние жизненного цикла

Block не знает [1]:

* где он используется
* в каком порядке
* в каком финальном тексте

=== Композиция (Composition) [[composition]]

Composition состоит из упорядоченного списка Fragment, где каждый фрагмент может быть [1]:

* **BlockRef** — ссылка на опубликованный Block с локальными переопределениями параметров
* **StaticText** — произвольный текст без параметров (markdown-разметка, XML-теги, логические связки), не имеет id и не версионируется отдельно от Composition
* **Separator** — типизированный разделитель (`newline`, `paragraph`, `hr`)

StaticText хранится как часть структуры Composition в поле `fragments_json`, не создаёт сущностей в Block storage [1].

=== StyleProfile [[style-profile]]

StyleProfile задаёт правила оформления текста и разделяется на два уровня:

==== StructuralStyle (структурный) [[structural-style]]
Детерминированные параметры, применяемые на этапе Template Expansion [1]:
* `output_format` — json | xml | markdown | plain
* `block_wrapper` — шаблон обёртки для каждого блока (например, `<final_instruction>{+{content}+}</final_instruction>`)
* `preamble/postamble` — статический текст до и после всей Composition
* `delimiters` — разделители между фрагментами

Применяется автоматически при рендеринге, не изменяет семантику текста [1].

==== SemanticStyle (семантический) [[semantic-style]]
Параметры, требующие переписывания текста:
* `tone`, `tense`, `target_language`, `person`

**Ограничение:** Не применяется автоматически. Требует явного вызова `normalize()` с подключенным Normalizer (LLM) [1]. При отсутствии Normalizer возвращается ошибка или игнорируется (в зависимости от режима strict/lenient).

== Расширенная структура Composition: Fragments [[fragments]]

Для повышения гибкости сборки текстов `Composition` поддерживает гибридную структуру, содержащую не только ссылки на переиспользуемые блоки, но и статические текстовые вставки (inline text), не требующие создания отдельных сущностей `Block` [1].

=== Типы фрагментов [[fragment-types]]

`Composition` состоит из упорядоченного списка `Fragment`, где каждый фрагмент может быть:

* **BlockRef** — ссылка на опубликованный `Block` с локальными переопределениями параметров [1]
* **StaticText** — произвольный текст без параметров (markdown-разметка, заголовки, разделители, логические связки)
* **Separator** — типизированный разделитель (`newline`, `paragraph`, `hr`), автоматически форматируемый под целевой стиль

=== Принципы использования StaticText [[static-text-principles]]

* Не имеет `id` и не версионируется — является частью структуры `Composition` [1]
* Не требует `RenderContext` для разрешения — текст используется as-is
* Не попадает в глобальное хранилище `Block` — не засоряет репозиторий одноразовыми строками [1]
* Сохраняется в `CompositionEntity` как часть JSON-структуры фрагментов

==== API работы с фрагментами [[fragment-api]]

[source,cpp]
----
class Composition {
public:
    // Добавить ссылку на блок с локальными параметрами [1]
    Fragment& add_block_ref(BlockId id, Version ver, Params local = {});

    // Добавить статический текст (заголовки, markdown, служебные теги)
    Fragment& add_static_text(std::string raw_text);

    // Добавить типизированный разделитель
    Fragment& add_separator(SeparatorType type);

    // Вставка/удаление/переупорядочивание
    void insert_fragment(size_t index, Fragment fragment);
    void remove_fragment(size_t index);
};
----

== Примеры использования StaticText [[static-text-examples]]

=== Пример 1: Markdown-структура документа
Разметка секций без создания отдельных блоков:

[source,cpp]
----
auto comp = engine.create_composition_draft("technical_spec");

comp.add_static_text("# Техническая спецификация\n\n");
comp.add_static_text("## Общие положения\n");
comp.add_block_ref({"role.analyst", "1.0"}, {{"domain", "backend"}});
comp.add_separator("paragraph");
comp.add_static_text("## Требования к производительности\n");
comp.add_block_ref({"constraint.latency", "2.1"}, {{"max_ms", "200"}});
comp.add_static_text("\n### Итоговые метрики\n");
comp.add_block_ref({"metrics.standard", "1.0"});

auto published = engine.publish_composition(comp.id());
// Результат: полноценный markdown-документ с заголовками и блоками
----

=== Пример 2: Логические связки между блоками
Текстовые переходы, объясняющие контекст:

[source,cpp]
----
auto report = engine.create_composition_draft("quarterly_report");

// Введение
comp.add_block_ref({"intro.template", "1.0"}, {{"quarter", "Q3"}});

// Логическая связка (не требует версионирования)
comp.add_static_text("\nНа основе представленных данных, ");
comp.add_static_text("можно сделать следующие выводы:\n\n");

// Основной контент
comp.add_block_ref({"analysis.financial", "3.2"});

// Разделитель секций
comp.add_separator("hr");
comp.add_static_text("**Примечание:** Данные актуальны на момент публикации.");
----

=== Пример 3: Условные разделители (без параметров в StaticText)
Комбинация параметризованных блоков и статических разделителей:

[source,cpp]
----
auto checklist = engine.create_composition_draft("deployment_checklist");

// Шапка с параметрами
comp.add_block_ref({"header.deploy", "1.0"}, {{"env", "production"}});

// Статические пункты чек-листа (неизменяемые инструкции)
comp.add_static_text("1. Проверить подключение к базе данных\n");
comp.add_static_text("2. Выполнить миграции\n");
comp.add_block_ref({"migration.warning", "2.0"}); // Параметризованный блок предупреждения
comp.add_static_text("3. Перезапустить сервисы\n");

// Разделитель перед итогами
comp.add_separator("newline");
comp.add_static_text("---\n**Важно:** В случае ошибки откат выполняется автоматически.");
----

=== Пример 4: Многоязычные разделители без LLM
Использование статического текста для языкоспецифичных конструкций, не требующих параметризации [1]:

[source,cpp]
----
// Для русской локали
auto ru_comp = engine.create_composition_draft("user_agreement_ru");
ru_comp.add_block_ref({"legal.header", "1.0"});
ru_comp.add_static_text("\nУважаемый пользователь,\n\n");
ru_comp.add_block_ref({"legal.body", "1.0"});
ru_comp.add_static_text("\nС уважением,\nКоманда сервиса");

// Для английской локали (без создания новых Block)
auto en_comp = engine.create_composition_draft("user_agreement_en");
en_comp.add_block_ref({"legal.header", "1.0"}); // Тот же блок, если шаблон универсален
en_comp.add_static_text("\nDear User,\n\n");
en_comp.add_block_ref({"legal.body", "1.0"});
en_comp.add_static_text("\nBest regards,\nService Team");
----

=== Пример 5: JSON/XML обёртки
Статический текст как синтаксические конструкции формата:

[source,cpp]
----
auto api_prompt = engine.create_composition_draft("json_schema_prompt");

comp.add_static_text(R"({
  "role": ")");
comp.add_block_ref({"role.developer", "2.0"});
comp.add_static_text(R"(",
  "constraints": [)");
comp.add_block_ref({"constraint.json_only", "1.0"});
comp.add_static_text(R"(],
  "context": ")");
comp.add_block_ref({"context.data", "1.0"}, {{"source", "api"}});
comp.add_static_text(R"("
})");
----

=== Пример 6: Комментарии и инструкции для LLM
Вставка мета-инструкций, не требующих параметризации:

[source,cpp]
----
auto prompt = engine.create_composition_draft("code_review");

comp.add_static_text("<!-- Начало системной инструкции -->\n");
comp.add_block_ref({"role.code_reviewer", "1.0"});
comp.add_separator("newline");
comp.add_static_text("<!-- Контекст задачи -->\n");
comp.add_block_ref({"task.description", "1.0"}, {{"ticket_id", "PROJ-123"}});
comp.add_separator("newline");
comp.add_static_text("<!-- Код для анализа (не изменять) -->\n```python\n");
comp.add_block_ref({"code.input", "1.0"});
comp.add_static_text("\n```\n<!-- Конец контекста -->");
----

=== Пример 7: Версионирование структуры vs контента
Демонстрация, что `StaticText` изменяется в Draft `Composition`, не создавая новых версий `Block` [1]:

[source,cpp]
----
// Создаем Draft
auto comp = engine.create_composition_draft("changelog");

// Добавляем статический заголовок (можно изменить без публикации нового Block)
comp.add_static_text("# Changelog v2.0\n\n## Новые функции\n");

// Добавляем блоки с фичами (версионированы отдельно)
comp.add_block_ref({"feature.auth", "1.0"});
comp.add_block_ref({"feature.cache", "2.0"});

// Позже редактируем Draft: меняем статический текст (заголовок)
// Это не требует создания новых версий BlockRef [1]
comp.clear_fragments(); // или редактирование конкретного фрагмента
comp.add_static_text("# Release Notes v2.1\n\n## Основные изменения\n");
// BlockRef остаются теми же версий 1.0 и 2.0

auto published = engine.publish_composition(comp.id());
----

=== Пример 8: Повторяющиеся паттерны без параметров
Шаблонные фразы, не требующие параметризации:

[source,cpp]
----
auto dialog = engine.create_composition_draft("support_dialog");

comp.add_block_ref({"greeting.support", "1.0"}, {{"user_name", "Алексей"}});
comp.add_static_text("\nСпасибо за обращение в службу поддержки.\n");
comp.add_block_ref({"problem.analysis", "1.0"});
comp.add_separator("paragraph");
comp.add_static_text("Если у вас остались вопросы, пожалуйста, дайте нам знать.\n");
comp.add_static_text("С уважением,\nКоманда поддержки");
----

=== Пример 9: Комбинированный рендеринг (Runtime + Static)
`RenderContext` влияет только на BlockRef, StaticText игнорирует контекст [1]:

[source,cpp]
----
auto comp = engine.create_composition_draft("dynamic_report");

comp.add_static_text("=== ОТЧЁТ ===\n"); // Неизменно
comp.add_block_ref({"date.generated", "1.0"}); // Может использовать {{system.date}}
comp.add_static_text("\nСтатус: "); // Неизменно
comp.add_block_ref({"status.current", "1.0"}, {{"priority", "high"}}); // Параметризован
comp.add_static_text("\n--- Конец отчёта ---"); // Неизменно

RenderContext ctx;
ctx.params = {{"priority", "low"}}; // Влияет только на status.current
// StaticText остается без изменений, не требует параметров
----

=== Пример 10: Отказ от использования StaticText (альтернатива)
Демонстрация избыточности создания Block для простого текста [1]:

[source,cpp]
----
// ❌ Плохо: создаем Block для одноразовой строки (засоряет Storage)
auto temp_block = engine.create_block_draft("temp.header",
    Template("### Заголовок раздела"), Params());
auto temp_pub = engine.publish_block(temp_block.id());
comp.add_block_ref({temp_pub.id(), temp_pub.version()});
// Теперь в Storage есть сущность, которая нигде больше не используется [1]

// ✅ Хорошо: используем StaticText
comp.add_static_text("### Заголовок раздела");
// Не создает сущности в Block storage, хранится только в Composition [1]
----

=== Мотивация [[fragment-motivation]]
Использование `StaticText` позволяет избежать создания избыточных сущностей `Block` для [1]:
* Markdown-разметки и заголовков
* Одноразовых переходных фраз ("Итого:", "Внимание:")
* Синтаксических конструкций (скобки, кавычки, разделители JSON/XML)
* Языкоспецифичных шаблонов, не требующих параметризации

Это соответствует принципу разделения ответственности: `Block` отвечает за переиспользуемую параметризованную логику, а `Composition` — за структуру и оформление конкретного текста [1].


== Жизненный цикл сущностей [[lifecycle]]

=== Состояния [[lifecycle-states]]
* *Draft* — редактируемая версия [1]
* *Published* — зафиксированная неизменяемая версия [1]
* *Deprecated* — устаревшая, но доступная для чтения [1]

=== Правила переходов [[lifecycle-rules]]
* Draft можно изменять без ограничений [1]
* Published версии неизменяемы, адресуются по версии и могут использоваться <<prompt-server,Prompt Server>> [1]
* Любое изменение Published сущности создаёт новую версию через Draft [1]
* Deprecated версии не удаляются и не используются по умолчанию [1]

== Этапы обработки текста [[text-processing]]

[source]
----
Blocks
↓
Template Expansion
↓
Raw Text
↓
(optional) Normalization
↓
Final Text
----

=== Разворачивание шаблонов (Template Expansion) [[template-expansion]]
Детерминированная стадия [1]:

* подстановка параметров
* разворачивание словарей
* выбор вариантов

=== Normalization (опциональный, явный вызов) [[normalization]]

Отдельный этап, который [1]:
* Вызывается явно через `engine.normalize(text, semantic_style)`
* Может быть реализован через LLM
* Может быть полностью отключён

**Запрещено:** Неявный вызов Normalization внутри `render()` при наличии SemanticStyle. Это нарушает детерминизм и контроль над текстом [1].

[source,cpp]
----
// Правильный порядок:
auto rendered = engine.render(comp_id, ctx); // Детерминировано [1]
auto normalized = engine.normalize(rendered, style.semantic()); // Явный вызов LLM
----

== Поддержка языков и времени (i18n) [[i18n]]

=== Стратегия многоязычности [[i18n-strategy]]

Без LLM поддерживаются только [1]:
* **Словари терминов** — параметризация Block через `{+{greeting}+}`, `{+{instruction}+}`
* **Языкоспецифичные версии Block** — явное создание `greeting.ru`, `greeting.en` как отдельных сущностей
* **Нейтральные формулировки** (код, markdown)

**Запрещено:** Автоматический перевод или адаптация текста Block под целевой язык через SemanticStyle без явного вызова Normalization [1].

=== Ограничения без LLM [[i18n-limitations]]
Следующие функции не поддерживаются без привлечения LLM [1]:

* полноценный перевод текста
* автоматическое изменение времени
* сохранение стиля при переписывании

== Модель хранения (Storage Model) [[storage-model]]

Storage является инфраструктурным слоем и полностью отделён от бизнес-логики движка [1].

=== Принципы хранения [[storage-principles]]

* движок не зависит от конкретной БД [1]
* работа идёт только через репозитории [1]
* формат хранения не является частью публичного API [1]

=== Проекты и пространства имён [[namespaces]]
Минимальная адресация сущностей [1]:

* `project_key`
* `entity_type`
* `entity_id`
* `version`

Разделение по проектам является обязательным требованием для <<prompt-server,Prompt Server>> [1].

=== Режимы доступа (Read/Write) [[access-modes]]
Допускается разделение режимов [1]:

* **read-write** — используется UI и редакторами
* **read-only** — используется Prompt Server

Важно: Prompt Server никогда не работает с Draft-данными [1].

=== Версионирование [[versioning]]
Storage обязан поддерживать [1]:

* хранение версий и доступ к конкретной версии
* неизменяемость опубликованных сущностей

== Публикация и синхронизация [[publishing]]

=== Операция публикации [[publishing-operation]]
Publishing — явная необратимая операция [1]:

* фиксирует текущую Draft-версию
* создаёт Published-версию
* делает её доступной для <<prompt-server,Prompt Server>>

=== Модель синхронизации [[sync-model]]

* UI работает с Workspace (Draft) [1]
* Prompt Server читает только Published Storage [1]
* Обновление промта = публикация новой версии [1]
* Старые версии продолжают обслуживаться [1]

== Интерфейсы: CLI и Terminal UI [[interfaces]]

=== CLI (обязательный компонент) [[cli]]
CLI является *reference client* для Text Engine и архитектурным якорем проекта [1].

Назначение CLI [1]:

* проверка полноты Core API
* инженерное использование без GUI
* автоматизация (скрипты, CI)
* публикация и инспекция промтов

CLI не предназначен для визуального редактирования или замены GUI [1].

=== Terminal UI (TUI) [[tui]]
TUI рассматривается как *UX-обёртка* над CLI, а не как самостоятельный клиент [1].

Архитектурные ограничения TUI [1]:

* использует CLI или Core API напрямую
* не содержит логики рендеринга
* не управляет версиями напрямую
* не имеет собственного слоя хранения

Статус реализации: не является обязательным для MVP, может быть реализован после стабилизации CLI [1]. Отсутствие TUI не блокирует развитие <<prompt-server,Prompt Server>> или UI [1].

== Сервер промптов (Prompt Server) [[prompt-server]]

=== Назначение и ограничения [[prompt-server-purpose]]
Prompt Server — сервис дистрибуции и разрешения промтов [1]:

* не редактирует данные
* не генерирует новые версии
* не содержит логики рендеринга (делегирует <<renderer,Renderer>>)

=== Протокол взаимодействия [[prompt-server-protocol]]
Запрос клиента [1]:

* `project_key`
* `prompt_name` (или путь)
* параметры (опционально)

Ответ сервера [1]:

* финальный текст
* метаданные (версия, язык, стиль)

Внутренняя логика: разрешает <<composition,Composition>>, загружает зафиксированные версии <<block,Block>> через <<storage-model,Storage>> (только <<access-modes,read-only>> режим), вызывает <<renderer,Renderer>> [1].

=== Стратегия разрешения параметров [[parameter-resolution]]
Параметры разрешаются по иерархии (от менее приоритетных к более приоритетным) [1]:

1. **Block Defaults** — параметры, определённые в Draft/Published <<block,Block>>
2. **Composition Local Override** — `local_params` в `BlockRef` внутри <<composition,Composition>>
3. **Runtime Context** — параметры, переданные в `RenderContext` при вызове `render()`

=== Строгие vs опциональные параметры [[strict-vs-default]]
* **Default**: Block содержит `default_params`. Если параметр не переопределён, используется значение по умолчанию [1].
* **Strict**: Block содержит `params_schema` с флагом `required`. Отсутствие значения вызывает ошибку (соответствует принципу <<no-magic,"нет магии">>).

== Схема хранения (ObjectBox) [[storage-schema]]

=== Сущности [[storage-entities]]
*BlockEntity* — хранит версии блоков с полями: `block_id`, `project_key`, `state` (0=Draft, 1=Published, 2=Deprecated), версионирование, шаблон, параметры [1].

*CompositionEntity* — хранит структуру композиции: `composition_id`, `project_key`, `block_refs_json` (массив ссылок на блоки с версиями), целевой язык и стиль [1].

[, cpp]
----
struct CompositionEntity {
// ... существующие поля ...

    // JSON массив Fragment вместо простого block_refs_json:
    // [
    //   {"type": "block_ref", "block_id": "role.system", "version": "1.2", "local_params": {...}},
    //   {"type": "static_text", "content": "### Итоговые выводы\n\n"},
    //   {"type": "separator", "kind": "paragraph"}
    // ]
    std::string fragments_json;
};
----

=== Паттерн Repository [[repository-pattern]]
`IBlockRepository` и `ICompositionRepository` — чистые интерфейсы (ports), не зависящие от ObjectBox [1]. Реализация `ObjectBoxBlockRepository` содержит специфику хранилища [1].

== Пример использования (Library Usage) [[workflow-example]]
Полный цикл: инициализация хранилища, создание Draft <<block,Block>> с параметрами по умолчанию, <<publishing-operation,публикация>>, создание <<composition,Composition>> с переопределением параметров, рендеринг с Runtime-контекстом [1].

== Критические ограничения реализации [[critical-constraints]]

. **Draft Parameters**: становятся неизменяемыми при переходе в Published [1]
. **Immutable Published**: сущность становится read-only, изменение создаёт новую версию [1]
. **Composition BlockRef**: обязательное указание версии (`version`) для воспроизводимости [1]
. **Rendering Failure**: отсутствие параметра возвращает `Error::MissingParam` вместо магической подстановки (соответствует <<control-over-convenience,контролю над текстом>>) [1]

----
Prompt Server [1]:
* Работает только с Published сущностями
* Применяет StructuralStyle автоматически (детерминировано)
* **Не применяет SemanticStyle автоматически** — возвращает текст как есть, либо возвращает ошибку, если требуется Normalization, но не выполнен явный вызов
----

== Модель доступа Prompt Server [[prompt-server-access]]
Prompt Server работает только с Published сущностями [1]:

* Запрос: `GET /prompts/blog.golang?v=1.0&audience=junior`
* Обработка: `engine.render("blog.golang", v1.0, {audience: "junior"})`
* Ответ: финальный текст + метаданные использованных блоков

Prompt Server никогда не вызывает `publish_*` или `save_draft` методы (<<access-modes,read-only режим>>) [1].
